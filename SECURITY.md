# Security Policy

## Reporting a Vulnerability

If you find a security vulnerability in ClearClaw, **please do not open a
public GitHub issue.**

Email: **security@clearclaw.dev**

Include:
- Description of the vulnerability
- Steps to reproduce
- Potential impact
- Any suggested fixes

You will receive an acknowledgement within 48 hours and a resolution timeline
within 5 business days.

## What ClearClaw Sends to the Cloud

ClearClaw connects to a Supabase instance that **you own and control**.
There is no "ClearClaw central server" receiving your data unless you
explicitly configure one.

### Exact fields sent per event (cloud mode only)

```
source        "claude_code" or "openclaw"
type          event type enum: file_read, file_write, command_exec,
              network_request, tool_call, session_start, session_end, heartbeat
sessionId     opaque session identifier generated by the agent
timestamp     ISO-8601 datetime
payload       SANITIZED metadata — see below
plainEnglish  human-readable translation (e.g. "read file /app/index.ts")
anomalyScore  float 0.0–1.0
anomalyReason short string describing anomaly trigger, or null
```

### Payload sanitization

Before any data is inserted, `sanitizePayload()` strips the following keys
(case-insensitive):

```
content, output, stdout, stderr, data, body, text,
file_content, filecontent, result, response, html, raw,
secret, password, token, key, credential, value
```

Source: `packages/daemon/src/supabase.ts` → `sanitizePayload()`

### Heartbeat events

Every 30 seconds the daemon sends a heartbeat. It contains only:

```json
{
  "source": "claude_code",
  "type": "heartbeat",
  "sessionId": "daemon-heartbeat",
  "payload": { "version": "0.1.0" }
}
```

No user data is included in heartbeats.

## What ClearClaw NEVER Sends

- **File contents** — the hook receives file paths, not file contents
- **Command output** — stdout, stderr, and exit codes are never captured
- **API keys, tokens, passwords, or secrets**
- **Environment variable values**
- **Any fields stripped by `sanitizePayload`** (see above)
- **User PII** beyond what the AI agent itself logged at WARN+ level

## How to Verify Traffic Yourself

### Option 1: Wireshark / tcpdump

```bash
# Capture all traffic leaving the daemon process
sudo tcpdump -i any -A -s 0 'host your-project.supabase.co' 2>/dev/null | grep -v '^$'
```

You should see only the fields listed in the "Exact fields sent" section above.
No file contents, no command output.

### Option 2: Proxy through mitmproxy

```bash
# Install mitmproxy
brew install mitmproxy

# Run the transparent proxy
mitmproxy --mode transparent --showhost

# Set system proxy to 127.0.0.1:8080, then start the daemon
node packages/daemon/dist/index.js
```

Inspect each request body in the mitmproxy UI to confirm field contents.

### Option 3: Read the source

The complete network layer is in one file:
`packages/daemon/src/supabase.ts`

The `streamEventToSupabase` function calls `sanitizePayload` on the event
before inserting. No other network calls exist in the daemon.

## Local-Only Mode (No Network Calls)

Set `LOCAL_ONLY=true` or add `{"localOnly": true}` to `~/.clearclaw/config.json`.

In local-only mode:
- All Supabase imports are bypassed
- Events are written to `~/.clearclaw/events.jsonl`
- Events are printed to the terminal
- The Unix socket server still runs (so hooks still fire)
- Translation and anomaly detection still run
- **Zero network calls are made**

Verify with:
```bash
LOCAL_ONLY=true node packages/daemon/dist/index.js &
# Watch the JSONL file
tail -f ~/.clearclaw/events.jsonl
```

## Architecture

```
┌─────────────────────────────────────────────────────┐
│                   Your Machine                       │
│                                                      │
│  Claude Code ──hook.sh──► Unix socket               │
│                            /tmp/clearclaw.sock      │
│                                │                     │
│  OpenClaw logs ─fs.watch──►   │                     │
│  ~/.openclaw/events.log        │                     │
│                                ▼                     │
│                        ┌──────────────┐             │
│                        │  daemon      │             │
│                        │  - parse     │             │
│                        │  - translate │             │
│                        │  - score     │             │
│                        │  - sanitize  │             │
│                        └──────┬───────┘             │
│                               │                     │
│                    ┌──────────┴────────────┐        │
│                    │                       │        │
│               LOCAL_ONLY=true        cloud mode     │
│                    │                       │        │
│                    ▼                       ▼        │
│           ~/.clearclaw/         YOUR Supabase      │
│           events.jsonl           (you own it)       │
└──────────────────────────────────────────────────── ┘
                                        │
                                        ▼
                              Next.js Dashboard
                              (your deployment)
```

**Key properties:**
- The daemon runs entirely on your machine
- The Supabase instance belongs to you (you created it, you hold the keys)
- No data routes through ClearClaw servers
- Local-only mode requires zero third-party accounts

## Supply Chain

Dependencies are locked via `pnpm-lock.yaml`. The daemon has only two
production dependencies:

- `@supabase/supabase-js` — official Supabase client
- `zod` — schema validation

Audit them:
```bash
pnpm audit
```
